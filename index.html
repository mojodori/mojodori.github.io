// 설계 의도 — 왜 이렇게 구성했는가

각 기술 선택에는 이유가 있습니다. "쓸 수 있어서"가 아니라 "이게 더 나은 선택이어서" 골랐습니다.

▸왜 Terraform + Ansible을 같이 썼는가
  • 대안: 콘솔에서 수동으로 인프라 생성
  • 이유: Terraform은 "무엇을 만들지(프로비저닝)", Ansible은 "그 안에 무엇을 설치할지(구성)" — 역할이 다름. 코드로 관리하면 인프라 재현성 100% 확보, 팀원 누구나 동일한 환경을 즉시 재현 가능
  • 로컬 state 사용의 한계(팀 충돌 위험)도 인지 — 실무라면 S3 backend + DynamoDB 락 적용 필요
▸왜 HPA 레플리카 수를 3개로 정했는가
  • 대안: 감으로 결정하거나 "넉넉하게 4~5개"
  • 이유: K6로 동시 사용자를 점진적으로 늘리며 부하를 주고 Grafana로 Pod별 CPU/메모리를 실시간 모니터링 — 레플리카 2개일 때 응답 지연, 4개일 때 리소스 대비 효율 저하 지점을 직접 측정해서 3개로 결정
  • "감이 아닌 데이터 기반 인프라 의사결정" 경험
▸왜 Route 53 + ACM + ALB 조합을 썼는가
  • 대안: 직접 인증서 발급 · 단순 IP 접속
  • 이유: ACM은 SSL 인증서를 무료로 자동 갱신해서 수동 관리 부담이 없음. ALB에서 HTTP→HTTPS 리디렉션을 한 곳에서 처리해 각 서버가 TLS를 신경 쓸 필요가 없어짐. Route 53으로 도메인 관리를 AWS 생태계 안에서 일원화
▸왜 보안 그룹을 계층별로 나눴는가
  • 대안: 보안 그룹 하나로 모든 서버 트래픽 통합 관리
  • 이유: Web(80/443) → WAS(8080) → DB(3306) 순서로만 통신 가능하도록 계층 분리 — DB는 인터넷에서 직접 접근 불가, WAS를 통해서만 도달 가능. 이 경험이 이후 홈랩 UFW 화이트리스트 설계로 이어짐
// 트러블슈팅
▸개발팀과 배포 일정 불일치
  • 문제: 개발팀 코드 완성 시점과 인프라 배포 일정이 맞지 않아 배포가 지연되거나, 인프라 환경과 코드가 호환되지 않는 문제가 반복됨
  • 원인: 개발팀과 인프라팀의 작업 시간이 달랐고, 오프라인 미팅이 어려운 환경이라 실시간 소통이 부족했음
  • 해결: 슬랙으로 실시간 소통 채널 구축 + 인프라팀 내부 하루 두 번 스탠드업 미팅으로 일정 지속 동기화. 전체 일정 재조정 회의를 거쳐 개발-배포 사이클을 명확히 정의한 이후 일정 충돌 감소
  • 이 경험으로 사전 일정 계획과 지속적인 소통의 중요성을 체감

// 설계 결정과 트레이드오프

단순히 "무엇을 설치했는가"가 아니라 "왜 이렇게 설계했는가"를 정리했습니다. 각 결정에는 포기한 대안과 그 이유가 있습니다.

▸왜 Bastion Host를 단일 진입점으로 뒀는가
  • 대안: 각 서버에 직접 외부 접속을 열어두는 방식
  • 이유: 외부 노출 지점을 9개에서 1개로 축소 — 공격 표면 최소화(Attack Surface Reduction)
  • SSH 키 인증·비밀번호 차단·로그인 3회 제한을 Bastion 한 곳에만 적용해도 전체 인프라가 동일 수준으로 보호됨
▸왜 WEB과 WAS를 분리했는가 (리버스 프록시)
  • 대안: DVWA가 도는 WAS에 사용자가 직접 접속
  • 이유: WAS의 실제 IP·포트 은닉, HTTPS·보안헤더를 WEB 한 곳에서 일괄 처리
  • 실무의 ALB/Nginx 리버스 프록시 패턴을 그대로 재현
▸왜 DB 계정을 용도별로 분리했는가
  • 대안: root 계정 하나로 모든 서버가 DB 접속
  • 이유: wasuser는 homelab_db 전체 권한, logstash는 alerts 테이블 INSERT만 — 최소 권한 원칙(PoLP)
  • SIEM 서버가 뚫려도 logstash 권한이 INSERT로 제한돼 기존 데이터 조회·삭제 불가 → 피해 범위 한정
▸왜 alert를 MariaDB + Elasticsearch에 동시 저장했는가
  • 대안: Elasticsearch 단독 저장
  • 이유: MariaDB는 정형 쿼리(IP별 공격 횟수 등) · 장기 보관에 강하고, Elasticsearch는 전문 검색 · Kibana 시각화에 강함
  • 실제 SOC도 SIEM(분석)과 별도 DB(리포팅·컴플라이언스)를 함께 운영 — 이중 구조를 재현
▸왜 NAT로 단순화했는가 — VLAN/DMZ 분리를 포기한 이유
  • 원래 계획: pfSense/OPNsense로 DMZ-APP-DB 물리적 VLAN 분리
  • 막힌 이유: M2 Mac(ARM64) — pfSense 공식 이미지가 x86 전용이라 설치 자체가 불가능
  • 대안 선택: Bastion만 Bridged로 분리, 나머지는 NAT 내부에서 UFW로 논리적 접근 제어를 흉내냄
  • 물리적 분리가 불가능할 때 논리적 접근 제어로 보완한 트레이드오프 — 한계를 인지하고 다음 단계로 계획 중
// DB 계정 권한 & 방화벽 규칙 상세

계정·트래픽 모두 "필요한 만큼만, 명시적으로" 허용하고 나머지는 기본 차단합니다. 아래는 실제 적용한 정책 전문입니다.

▸DB 계정 권한 — wasuser · logstash 분리 (SQL 전문)
  • wasuser — WAS 전용 계정, homelab_db 전체 권한
  • CREATE USER 'wasuser'@'172.16.60.134' IDENTIFIED BY 'SecureDBPass!';
    GRANT ALL PRIVILEGES ON homelab_db.* TO 'wasuser'@'172.16.60.134';
  • logstash — SIEM 전용 계정, alerts 테이블 INSERT만
  • CREATE USER 'logstash'@'172.16.60.136' IDENTIFIED BY 'LogstashPass!';
    GRANT INSERT ON homelab_db.alerts TO 'logstash'@'172.16.60.136';
  • 설계 포인트 ① — %(전체) 대신 실제 IP 명시 → 다른 IP에서는 접속 자체 불가
  • 설계 포인트 ② — SIEM이 뚫려도 logstash 계정은 INSERT만 가능 → 기존 alert 조회·수정·삭제 불가, 피해 범위 한정
▸방화벽 규칙 — UFW 화이트리스트 전문
  • Bastion — 단일 진입점, 22번만 개방
  • sudo ufw allow 22/tcp
    sudo ufw enable
  • WEB — 외부 트래픽 진입점, 80/443만 개방
  • sudo ufw allow 80/tcp
    sudo ufw allow 443/tcp
    sudo ufw enable
  • DB — 출발지 IP를 명시해서 3306만 허용 (포트만 여는 것과 다름)
  • sudo ufw allow from 172.16.60.134 to any port 3306  # WAS에서만
    sudo ufw allow from 172.16.60.136 to any port 3306  # SIEM에서만
    sudo ufw enable
  • SIEM — Filebeat 수신용 5044번만 개방
  • sudo ufw allow 5044/tcp
  • 핵심: ufw allow 3306처럼 포트만 여는 게 아니라 from <IP>로 출발지까지 제한 — 같은 NAT 대역 안의 다른 서버도 명시되지 않으면 DB 접근 불가

▸구현 방법 상세
  • L3 스위치: GNS3 가상 스위치에서 VLAN 10/20/30/40/50 태깅, 802.1Q 트렁크로 라우터와 연결
  • 라우터/방화벽: OPNsense(또는 VyOS)에서 인터페이스별 VLAN 할당 + Inter-VLAN 라우팅 ACL 설정
  • 방화벽 정책: 기본 deny-all, VLAN 간 통신은 화이트리스트로만 명시 (예: DMZ→APP 8080, APP→DB 3306)
  • ATTACK VLAN 격리: Kali는 DMZ 방향 트래픽만 허용, APP·DB·MGMT로의 직접 접근은 라우터 단에서 차단 — 실제 침해 시나리오처럼 외부에서 내부로 침투하는 경로만 열어둠
  • 검증 방법: Kali에서 DB(192.168.30.x)로 직접 스캔 시도 → 차단 확인, DMZ 경유 시에만 APP까지 도달하는지 traceroute로 검증

각 구성 요소는 "그냥 흔히 쓰니까"가 아니라, 실제로 겪은 문제와 대안 비교를 거쳐 선택했습니다.

▸왜 Bastion을 통해서만 접근하게 했는가
  • 대안: 각 서버에 SSH를 직접 노출
  • 이유: 진입점을 하나로 좁히면 공격 표면이 줄고 접근 기록을 한 곳에서 감사할 수 있음. 실제 nmap 스캔에서 was-01·web-lb가 외부(맥북) 기준 전 포트 filtered로 관측되어 검증됨
▸왜 DB 트랜잭션 락 대신 Redis 원자연산을 썼는가
  • 대안: DB 트랜잭션 + 행 잠금(SELECT FOR UPDATE)
  • 이유: DB 락은 동시 요청이 몰릴수록 대기 큐가 길어져 응답이 느려짐. Redis는 인메모리 단일 스레드라 원자적 명령 하나로 즉시 처리 가능 — K6 100건 동시요청에서 평균 응답 30ms대를 유지하며 중복 0건 달성
▸왜 대기열이 "느리게 처리"가 아니라 "초과분 즉시 거절" 방식인가
  • 대안: 초과 요청을 큐에 쌓아 순서대로 처리
  • 이유: 큐에 쌓으면 대기 중인 연결이 계속 늘어 서버 자원이 고갈됨(실측: 대기열 없이 최대 응답 60초). 초과분을 429로 즉시 거절하면 서버는 항상 감당 가능한 용량 안에서만 일해 가용성이 유지됨
▸왜 DB Read Replica를 뒀는가
  • 대안: DB 한 대가 조회·쓰기 모두 처리
  • 이유: 티켓 오픈 시 좌석 조회 트래픽이 예약 확정(쓰기)보다 수십~수백 배 많음. 조회를 Replica로 분리하면 진짜 중요한 쓰기 트랜잭션이 조회 부하 때문에 밀리지 않음
▸왜 Suricata를 별도 서버가 아닌 queue-gate에 설치했는가
  • 대안: 독립된 모니터링 서버에 설치(최초 계획)
  • 이유: 별도 서버는 실제 패킷이 지나가지 않아 아무것도 탐지하지 못한다는 걸 직접 겪음. IDS는 반드시 트래픽 경로 위에 있어야 동작한다는 걸 실습으로 확인 후 구조를 변경

프론트엔드 회원가입/로그인 화면과 90일 비밀번호 만료 흐름은 데모로 구현했으며, 아래는 실제 백엔드 연동 시 적용할 설계입니다 (현재 API 서버는 좌석예약 로직만 연동된 상태입니다).

  1. 사용자가 이메일 · 비밀번호(10자 이상) 입력
  2. 서버가 비밀번호를 bcrypt(cost 12)로 해시 — 원문은 저장도, 전송 로그에도 남기지 않음
  3. members 테이블에 password_hash와 password_changed_at(현재시각) 저장
  4. 로그인 시점마다 password_changed_at과 현재시각 비교, 90일 초과 시 세션 발급 대신 재설정 화면으로 전환
  5. 재설정 시 최근 비밀번호 이력(password_history)과 대조해 재사용 차단
CREATE TABLE members (
  id INT AUTO_INCREMENT PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  password_changed_at DATETIME NOT NULL DEFAULT NOW(),
  password_history JSON,
  failed_login_count INT DEFAULT 0
);

const hash = await bcrypt.hash(password, 12);
await db.query(
  'INSERT INTO members (email, password_hash, password_changed_at) VALUES (?, ?, NOW())',
  [email, hash]
);

  • Suricata 최초 별도 모니터링 서버에 배치 → 실제 트래픽 경로 밖이라 탐지 불가 → 트래픽이 지나가는 queue-gate에 재설치하여 해결
  • Suricata 설정파일의 인터페이스명(eth0→ens160)·용량 단위(KiB→kb) 오류를 로그 분석으로 직접 수정
